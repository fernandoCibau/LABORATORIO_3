Preguntas sobre HTTP

  1. ¿Que protocolo binario está montado el protocolo HTTP?

    Respuesta :El protocolo HTTP esta montado sobre el protocolo tcp/ip( Protocolo de Control de Transmision ) este es el encargado de espabrecer la conexiones "Cliente / Servidor", La misma se establece por un enlace de tres vias Handshaking ()

    1 - EL Host/Cliente inicia la conversacion solicitando una comunicacion al servidor Enviando un segmento SYN.

    2 - El Servidor responde a la solicitd cin otra señal (SYN-ACK)., "El ACK confirma que el servidor ha recibido el SYN"

    3 - En este ultimo paso, el cliente reconoce la respuesta del servidor, y ambos crean una conexion estable para comenzar el proceso real de transferencia de datos


  2. ¿Cuáles son los clientes http y los servidores http mas utilizados?

    Respuesta : Aquí te presento algunos de los servidores web más utilizados:

    1 - Nginx : Nginx es un servidor web de código abierto y alto rendimiento. Es ampliamente utilizado para gestionar sitios web de alto tráfico. Su éxito radica en su capacidad para manejar grandes cantidades de conexiones simultáneas. Grandes plataformas como Netflix y Pinterest confían en Nginx.

    2 - Apache HTTP Server : Apache, también conocido como Apache HTTP Server, es uno de los servidores web más populares del mundo. Es gratuito, de código abierto y ha sido fundamental en el desarrollo web durante décadas.

    3 - LiteSpeed : LiteSpeed es otro servidor web que se destaca por su rendimiento y eficiencia. Aunque no es tan ampliamente utilizado como Nginx o Apache, es una opción sólida para sitios web de alto rendimiento.

    4 - Microsoft-IIS : El Servidor de Información de Internet (IIS) de Microsoft es comúnmente utilizado en entornos Windows. Aunque su cuota de mercado es menor, sigue siendo relevante en el ámbito empresarial.

    5 - OpenResty : OpenResty es una plataforma basada en Nginx que permite la ejecución de código Lua en el servidor web. Es especialmente popular para aplicaciones web dinámicas y APIs.

    * En cuanto a los clientes HTTP : Los navegadores web son los más comunes. Algunos de los navegadores más utilizados incluyen Google Chrome, Mozilla Firefox, Microsoft Edge y Safari. Estos clientes envían solicitudes HTTP a los servidores web para obtener páginas web y recursos


  3. ¿Que verbos admite un comando en el requerimiento HTTP?

    Respuesta : Los métodos de petición HTTP (también conocidos como verbos HTTP) indican la acción que se desea realizar sobre un recurso específico. A continuación, te presento los métodos más comunes:

    1 - GET : Solicita una representación de un recurso específico. Las peticiones con el método GET solo recuperan datos.

    2 - POST : Envía una entidad a un recurso específico, causando cambios en el estado o efectos secundarios en el servidor.

    3 - PUT : Reemplaza todas las representaciones actuales del recurso de destino con la carga útil de la petición.

    4 - DELETE : Borra un recurso específico.

    5 - HEAD : Similar a GET, pero sin el cuerpo de la respuesta. Se utiliza para obtener metadatos de la cabecera.

    6 - CONNECT : Establece un túnel hacia el servidor identificado por el recurso.

    7 - OPTIONS : Describe las opciones de comunicación disponibles para el recurso de destino.

    8 - TRACE: Realiza una prueba de bucle de retorno de mensaje a lo largo de la ruta al recurso de destino.

    9 - PATCH : Aplica modificaciones parciales a un recurso.

    * Estos métodos son fundamentales para la comunicación efectiva entre clientes y servidores en el protocolo HTTP.


  4. ¿Que contenido lleva el body de un requerimiento HTTP?

    Respuesta : El cuerpo de una solicitud HTTP contiene datos específicos que se envían al servidor. La estructura y el contenido del cuerpo varían según el tipo de solicitud y el formato de datos utilizado

    Ejemplos :

    Las solicitudes GET : Generalmente no incluyen un cuerpo en la petición. Los datos se transmiten a través de los parámetros de la URL (por ejemplo, https://mi-sitio.com/recurso?parametro1=valor1¶metro2=valor2). En resumen, el cuerpo de una solicitud GET suele estar vacío.

    Las solicitudes POST : envían datos al servidor en el cuerpo de la solicitud. Estos datos pueden ser formularios HTML, JSON, XML o cualquier otro formato.
    Por ejemplo, al enviar un formulario de inicio de sesión, los campos como el nombre de usuario y la contraseña se incluirían en el cuerpo de la solicitud POST.

    Solicitudes PUT y PATCH : Estas solicitudes también incluyen datos en el cuerpo. En una solicitud PUT, el cuerpo contiene la representación completa del recurso que se va a actualizar. En una solicitud PATCH, el cuerpo contiene solo las modificaciones específicas que se deben aplicar al recurso existente.

    Solicitudes DELETE : Aunque las solicitudes DELETE no requieren un cuerpo, algunos servidores pueden aceptar un cuerpo opcional para proporcionar información adicional sobre la eliminación.


  5. ¿Que diferencia existe entre un URL y un URI?

    Respuesta :
    * URI (Identificador Uniforme de Recursos) Es una cadena que identifica un recurso web. Puede ser un nombre o una ubicación., mientras que una URL (Localizador Uniforme de Recursos) : Es un tipo específico de URI que solo proporciona la ubicación de un recurso en Internet .

    Ejemplo :

    URL ==> http://miempresa.midominio/
    URI ==> mirecurso.html


  6. ¿Como se almacena información relacionada con las respuestas http?

    Respuesta : La caché del navegador almacena temporalmente archivos de contenido de una página web en tu dispositivo local (como imágenes, CSS, JavaScript, etc.). Su objetivo principal es mejorar la velocidad de carga de las páginas web., Cuando visitas un sitio web, el navegador guarda estos recursos en la caché local, La próxima vez que vuelvas a esa página, el navegador utilizará los recursos almacenados en lugar de descargarlos nuevamente desde Internet.
    Ejemplos :
    * La cache guarda en una variable de protocolo(headers) con la fecha de la última vas que fue modificado el recurso del lado del servidor. Si la pagina vuelva a cargar el navegador solicita al servidor mediante esa fecha que está en el header, y el navegador controla si ese recurso fue modificado (si tiene otra fecha). Si es que no se modificó devuelve una variable MODIFIED = NOR MODIFIED, en esa caso el navegador usa el mismo recurso, caso contrario se vuelve a descargar.
    * La cache también almacena mediante mediate Etag’s y Hash’s, la diferencia es que usa un código. Eso resuelve fallas de fallas de actualizaciones por fechas.
    En resumen : La caché del navegador ayuda a que las páginas se carguen más rápido.


  7. ¿Que significa Virtual hosting?

    Respuesta : El Virtual Hosting (o anfitrión virtual) es una técnica que permite a un servidor web alojar múltiples sitios web en la misma máquina física. Esto se logra mediante la asignación de nombres de dominio o direcciones IP específicas a cada sitio web, lo que permite al servidor identificar y enrutar las solicitudes de manera adecuada.
    En resumen, el virtual hosting permite que varios sitios web compartan un mismo servidor físico, pero se comporten como si estuvieran en servidores separados. Cada sitio web tiene su propio espacio aislado y se puede acceder a través de su propio dominio o dirección IP.
    Por ejemplo : Si tienes varios sitios web (como “sitio1.com”, “sitio2.com” y “sitio3.com”), puedes alojarlos en el mismo servidor utilizando el virtual hosting. Cada sitio tendrá su propio espacio de almacenamiento, configuración y recursos, pero todos compartirán el mismo servidor físico.


  8. ¿Porque el HTTP se considera STATE LESS?

    Respuesta : HTTP se considera un protocolo sin estado o stateless porque no mantiene información de las transacciones anteriores. En el contexto de HTTP, el servidor trata cada solicitud como si fuera la primera vez que recibe una solicitud de ese cliente. No guarda ningún estado entre las transacciones durante una sesión. Esto significa que, si un cliente realiza dos solicitudes al servidor, este último no retiene ninguna información de la primera solicitud cuando procesa la segunda.
    Esta característica permite que los servidores HTTP sean más simples y rápidos, ya que no necesitan gestionar el estado de las interacciones con los clientes. Sin embargo, también significa que cualquier información adicional necesaria para manejar las solicitudes debe ser reenviada con cada solicitud. Por ejemplo, en una aplicación web que requiere inicio de sesión, el usuario debe enviar su información de autenticación con cada solicitud, o el sistema debe utilizar mecanismos como las cookies o tokens para mantener el estado a través de las solicitudes.


  9. ¿Como va a mejorar la velocidad de la WEB en poco tiempo?

    Respuesta : HTTP/2 apareció en 2015 para reemplazar al protocolo HTTP/1.1 e introducir numerosas mejoras. Esta versión del protocolo incorporó grandes cambios y mejoró significativamente la transferencia de datos, lo que también impulsó la aceleración de la velocidad de carga de las páginas web.

    A continuación, te presento algunas de las mejoras clave de HTTP/2 con respecto a HTTP/1.1:

    1. Eliminación de la información redundante :
    * HTTP/2 reduce la latencia y el consumo de recursos al eliminar información redundante en las solicitudes y respuestas.
    * Esto permite una comunicación más eficiente entre el cliente y el servidor.

    2. Multiplexación :
    * HTTP/2 resuelve el problema conocido como “Head-of-line blocking” de HTTP/1.1.
    * En HTTP/1.1, solo se podía usar una única solicitud por conexión TCP, lo que generaba congestión.
    * Con HTTP/2, se pueden resolver varias peticiones a la vez, evitando la congestión y mejorando la eficiencia.

    3. Compresión de encabezados :
    * HTTP/2 utiliza una técnica de compresión de encabezados para reducir el tamaño de las cabeceras.
    * Esto disminuye la sobrecarga de datos y mejora la velocidad de transferencia.

    4. Formato binario:
    * HTTP/2 utiliza un formato binario en lugar del formato de texto legible de HTTP/1.1.
    * El formato binario es más eficiente en términos de procesamiento y transmisión.


  10. ¿Como va a mejorar el comportamiento de una aplicación WEB en ambientes ruidosos o de débil conexión?

    Respuesta : HTTP/3 es la nueva versión del protocolo de transferencia de hipertexto (HTTP) y está basado en el protocolo UDP. puede mejorar el comportamiento de una aplicación web en ambientes ruidosos o con conexiones débiles.

    Es compuesto por :

    Protocolo UDP : HTTP/3 utiliza QUIC (Quick UDP Internet Connections) como su capa de transporte en lugar de TCP (Transmission Control Protocol). A diferencia de TCP, que es orientado a la conexión y espera confirmaciones de paquetes, UDP es un protocolo sin conexión. Esto significa que no hay esperas para confirmaciones, lo que reduce la latencia y mejora la velocidad de transferencia.

    Multiplexación nativa : QUIC permite la multiplexación de forma nativa. En HTTP/2, aunque se introdujo la multiplexación para transmitir múltiples recursos en una sola conexión TCP, aún estaba limitada por TCP. Si un paquete se perdía, toda la conexión se paralizaba. En HTTP/3, los paquetes perdidos solo afectan a las transmisiones específicas en las que se perdieron datos. Esto evita bloqueos y mejora la eficiencia.

    Menos sobrecarga : HTTP/3 elimina la sobrecarga de TCP. Al no depender de confirmaciones de paquetes, no hay necesidad de retransmitir paquetes perdidos en toda la conexión. Esto reduce la latencia y mejora la velocidad de carga.

    Mayor fiabilidad : Aunque UDP no incluye algunas características de TCP, QUIC compensa esto proporcionando retransmisión de paquetes y otros mecanismos de confiabilidad. Así, HTTP/3 ofrece una mayor fiabilidad que simplemente usar UDP.
